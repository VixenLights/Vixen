# Element Tags — Display Setup, Preview Setup, and Sequencer Workflow (VIX-2690)

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds. Maintain this document in accordance with `.agents/PLANS.md`.

This plan implements the *user-facing workflow* for Element Tags as specified in `docs/vix-2690-element-tags-workflow.md`. That specification document is the authoritative requirements source and this plan restates every fact from it needed to implement without re-reading it, but the spec's Decision Log carries the full history of why each choice was made — consult it if a decision here seems to need more justification than is repeated here. This plan builds entirely on top of VIX-3933's already-shipped, already-merged API and persistence layer (`docs/vix-3933-element-tags.md`, `docs/plans/vix-3933-element-tags-api.md`) — `IElementNode.Tags`, `ElementTagDefinition`, `ElementTagService`, `SystemConfig.Tags` persistence, and the three built-in tags (`Deprecated`, `Hidden`, `Prop`) with their default `DisplayColor` values (red `#FF0000`, black `#000000`, blue `#0000FF` respectively — added to VIX-3933's `BuiltInElementTags.CreateDefaults()` and already rebased into this branch, commit `1960a242c`) all already exist and are not touched by this plan except where explicitly noted.

## Purpose / Big Picture

Today, Vixen has no user-visible way to work with Element Tags at all. A developer could write code against `IElementNode.Tags`, but no menu, dialog, or visual indicator anywhere in Display Setup, Preview Setup, or the Sequencer lets an actual user assign a tag, see which elements are tagged, or have tag-driven behavior (like being blocked from sequencing a deprecated element) happen. This plan changes that: after it is complete, a user can right-click any element in Display Setup, Preview Setup, or the Sequencer and toggle `Deprecated`, `Hidden`, or `Prop` on it; see small colored dots after tagged elements' names everywhere they appear; hide `Hidden`-tagged elements from the Sequencer for the current session; be blocked (with visible feedback) from adding new effects to a `Deprecated`-tagged row in the Sequencer; get a one-time warning dialog when opening a sequence that has effects on now-deprecated elements; and open a small window to change what color each built-in tag is shown as.

Concretely, someone can see this working by: building the solution (`msbuild Vixen.sln -m -t:restore -t:Rebuild -p:Configuration=Debug`), launching `Vixen.Application`, opening Display Setup, right-clicking an element, choosing Tags > Deprecated, and observing a red dot appear after that element's name — then opening the Sequencer, seeing that same red dot next to that element's row label, dragging an effect onto that row and having the drop rejected, and finally closing and reopening the sequence to see the "elements tagged Deprecated have effects" notification dialog listing that element.

## Surprises & Discoveries

- Observation: `ToolStripMenuItem` has no `ThreeState` property — only `CheckBox` does. The plan's original design (written before implementation) assumed `ThreeState = true` was required alongside `CheckState` for the tri-state Tags checkmark; the first build after writing Milestone 2's code failed with `CS0117: 'ToolStripMenuItem' does not contain a definition for 'ThreeState'`. Removing that line and keeping only the `CheckState` assignment (set directly in `PopulateTagsMenu()`/`ToggleTagOnSelectedNodes()`) compiled cleanly and is sufficient — `CheckState.Indeterminate` renders its distinct glyph without any enabling flag.
  Evidence: `msbuild Vixen.sln -m -t:restore -t:Build -p:Configuration=Debug -v:minimal -clp:ErrorsOnly` initially printed exactly one error, `C:\Dev\Vixen\src\Vixen.Common\Controls\ElementTree.cs(934,6): error CS0117: 'ToolStripMenuItem' does not contain a definition for 'ThreeState'`; after removing the `ThreeState = true` initializer, the same command produced no errors, and `dotnet test src\Vixen.Tests\Vixen.Tests.csproj --nologo` reported `Passed! - Failed: 0, Passed: 118, Skipped: 0, Total: 118`.
- Observation: setting `CheckState.Indeterminate` compiled and worked at the data-model level, but the checkmark glyph looked identical to a fully `Checked` item — no visible distinction, even though `CheckState` was verifiably being set correctly. Root cause: this application does not use WinForms' default menu rendering; `ElementTree`'s `contextMenuStripTreeView.Renderer` is set to a custom `Common.Controls.Theme.ThemeToolStripRenderer` (`src\Vixen.Common\Controls\Theme\ThemeToolStripRenderer.cs`), and its `OnRenderItemCheck` override always drew the same `check_mark` resource image regardless of `CheckState`, only branching on `!e.Item.Enabled` (to draw a disabled variant). `ToolStripItem.Checked` is `true` for both `CheckState.Checked` and `CheckState.Indeterminate` (it is only `false` for `Unchecked`), so `OnRenderItemCheck` fires for both states identically — the custom renderer had no tri-state awareness at all, not just a Tags-menu-specific gap. This is a defect in shared theming code, not in Milestone 2's own menu logic.
  Evidence: user manually tested a 3-element selection where only one had the `Hidden` tag and reported "the tristate is not working" — the checkmark rendered the same in all cases. Reading `ThemeToolStripRenderer.OnRenderItemCheck` (`ThemeToolStripRenderer.cs`, then lines 235-251) confirmed the checkmark image and draw call were unconditional past the enabled/disabled branch.
- Observation: the Sequencer's row-label context menu is not owned by `TimedSequenceEditorForm.cs` as Milestone 3's original plan text guessed (that file's `_contextMenuStrip` is a different, unrelated timeline/effect-actions menu). It lives in `Common.Controls.Timeline.TimelineControl` (`src\Vixen.Common\Controls\TimeLineControl\TimelineControl.cs`), built fresh on every right-click inside `RowLabelContextMenuHandler`, subscribed to `RowLabel`'s right-click via a chain of relayed events: `RowLabel.OnMouseUp` → `RowLabel.RowContextMenuSelect` (instance event) → `Row.RowContextMenuSelectHandler` → `Row.RowLabelContextMenuSelect` (a *static* event on `Row`) → `TimelineControl.RowLabelContextMenuHandler`. The existing three row-level menu items (Collapse All Groups, Reset All Rows to Default Height, Reset Selected and Child rows to Default Height) confirmed the spec's expectation that this menu was purely row-layout actions before this change. Row selection is exposed via `TimelineControl.SelectedRows` (`IEnumerable<Row>`, backed by `grid.SelectedRows`), and each selected row's `ElementNode` is `row.Tag as ElementNode` — the same mapping pattern already used in `TimedSequenceEditorForm.cs` (e.g. line 3737, `var targetNode = (ElementNode) row.Tag;`).
  Evidence: `grep -n "RowContextMenuSelect" src -r` traced the full event chain across `RowLabel.cs` → `Row.cs` → `TimelineControl.cs`; `grep -n "RowLabelContextMenuHandler" TimelineControl.cs` located the existing three-item menu construction at line ~762.

## Decision Log

*(Record new implementation-time decisions here as work proceeds, in addition to the decisions already made and recorded in `docs/vix-2690-element-tags-workflow.md`'s own Decision Log, which this plan does not repeat verbatim.)*

- Decision: Milestone 2 (tag assignment via the shared `ElementTree` control) relies entirely on Display Setup's and Preview Setup's existing OK/close-gated `VixenSystem.SaveSystemConfigAsync()` calls and adds no save of its own. Milestone 3 (Sequencer tag assignment) always calls `VixenSystem.SaveSystemConfigAsync()` explicitly, since the Sequencer has no other path that would persist the change.
  Rationale: The user specified this directly — Display Setup's OK path and Preview Setup's normal close path already save and "should prevail" (i.e. not be duplicated or second-guessed by this plan), but any tag definition or assignment edited outside those two paths needs an explicit save so it isn't silently lost. Tracing the actual code (`VixenApplication.SetupDisplay()`/`SetupPreviews()`, `VixenApplication.cs` lines 957-1008) confirmed both paths follow the same shape: save on `DialogResult.OK`, `VixenSystem.ReloadSystemConfig()` (discarding unsaved changes) otherwise.
  Date/Author: 2026-07-02, requirements clarification pass with Jeff Uchitjil.

- Decision: Milestone 8's tag color editor (`ElementTagColorEditorWindow`) takes a generic `saveOnClose` constructor parameter and only calls `VixenSystem.SaveSystemConfigAsync()` on Save when it is true. Callers with no other save path (the Sequencer's Tags submenu, today) pass `true`; callers with their own OK/Cancel-gated save (Display Setup's and Preview Setup's Tags submenus) pass `false` and leave the color change to that host's own path, same as Milestone 2. This reverses an earlier version of this decision, which had the color editor always save regardless of host, and refines a narrower follow-up version that named the flag `invokedFromSequencer`.
  Rationale: The user first corrected the unconditional-save design: `SaveSystemConfigAsync()` saves the entire `SystemConfig`, not just tag data, so an unconditional save from inside a sub-dialog opened during an in-progress Display Setup or Preview Setup session would silently commit whatever other, unrelated, not-yet-confirmed changes the user has pending in that session too — there is no mechanism to save "just the tags" and leave the rest of a Display Setup/Preview Setup session's pending edits alone. The user then asked for the flag itself to be generalized rather than named after the Sequencer specifically, so the window's contract stays "should I persist my own changes" — reusable by any future caller with different save semantics — rather than baking in today's one caller that happens to need `true`.
  Date/Author: 2026-07-02, requirements clarification pass with Jeff Uchitjil.

- Decision: The Tags submenu's checkable items (Milestones 2 and 3) use `ToolStripMenuItem.CheckState` (`Checked`/`Indeterminate`/`Unchecked`), not a plain boolean `Checked`, so a multi-selection where only some nodes/rows have a tag renders a visually distinct, darker/muted checkmark instead of looking identical to "no node has it."
  Rationale: The user asked for this directly, specifying it should follow the same visual convention other applications already use for partial multi-selection state. `ToolStripMenuItem.CheckState.Indeterminate` is WinForms' built-in mechanism for exactly this and renders with the platform's own distinct/darker glyph, matching the same pattern Windows Explorer uses for partial-selection property checkboxes — no custom drawing needed. Click semantics were set to match that same precedent: clicking a partial (`Indeterminate`) checkmark completes it to fully `Checked` (assigns to all), rather than clearing it; only a fully `Checked` item clears to `Unchecked` on click.
  Correction made during Milestone 2 implementation: an earlier version of this decision (and the Context and Orientation/Milestone 2/Milestone 3 text) additionally referenced setting `ThreeState = true` on the `ToolStripMenuItem` before this would work. That property does not exist on `ToolStripMenuItem` (`ThreeState` is a `CheckBox`-only property) — the build failed with `CS0117` when first attempted. `CheckState` alone is sufficient: setting it directly to `Indeterminate`/`Checked`/`Unchecked` renders correctly with no other property needed, since the `Click` handler already sets `CheckState` explicitly rather than relying on any automatic click-cycling behavior. Fixed by removing the `ThreeState = true` initializer; confirmed by a clean solution build and the full test suite (118/118) passing afterward.
  Date/Author: 2026-07-02, requirements clarification pass with Jeff Uchitjil.

Vixen is a .NET 10 Winforms / WPF desktop app for sequencing animated light shows (see root `CLAUDE.md`). This plan touches three largely independent WinForms surfaces plus one new WPF window:

**Tri-state checkmarks for multi-selection.** When more than one node/row is selected and they disagree on whether they have a given tag (some have it, some don't), a plain on/off checkmark cannot represent that state accurately. `System.Windows.Forms.ToolStripMenuItem.CheckState` supports exactly this out of the box: it is a three-value `CheckState` (`Checked`/`Indeterminate`/`Unchecked`), and setting it to `Indeterminate` renders a visually distinct, filled/darker checkmark glyph rather than the normal bright checkmark, with no other property or custom drawing needed — this is the same tri-state visual Windows itself uses elsewhere for partial-selection checkboxes (e.g. Explorer's multi-file property dialogs). (There is a `CheckBox.ThreeState` property elsewhere in WinForms, but no equivalent property exists on `ToolStripMenuItem` — do not set it; `CS0117` results if you try. `CheckState` alone is sufficient here.) Both Milestones 2 and 3 use this: `CheckState.Checked` (normal-color checkmark) when *every* selected node/row has the tag, `CheckState.Indeterminate` (darker/muted checkmark) when *some but not all* have it, and `CheckState.Unchecked` (no checkmark) when *none* do. Since the `Click` handler always sets `CheckState` explicitly rather than relying on any automatic click-cycling behavior, the transition rule is simply: clicking while `Unchecked` *or* `Indeterminate` assigns the tag to every selected node/row (going to `Checked`); clicking while `Checked` removes the tag from every selected node/row (going to `Unchecked`) — i.e. `Indeterminate` behaves like `Unchecked` for the purpose of "what does a click do" (both result in "assign to all"), matching the same Explorer-style convention: clicking a partially-checked box completes the action for every item, it does not clear it; only a fully-checked box clears on click.

`ElementTagDefinition` (`src\Vixen.Core\Sys\ElementTagDefinition.cs`, `public sealed class`, namespace `Vixen.Sys`): `Id` (`Guid`, immutable), `Key` (`string`, immutable), `Name` (`string`, settable), `IsBuiltIn` (`bool`, immutable), `Description` (`string?`, settable), `DisplayColor` (`string?`, settable, e.g. `"#FFA500"`), `SortOrder` (`int`, settable).

`src\Vixen.Core\Sys\BuiltInElementTags.cs`: `DeprecatedKey`/`DeprecatedId`/`DeprecatedColor` (`"#FF0000"`), `HiddenKey`/`HiddenId`/`HiddenColor` (`"#000000"`), `PropKey`/`PropId`/`PropColor` (`"#0000FF"`), `CreateDefaults()` returns all three with `SortOrder` 0/1/2. Every profile — new or pre-existing — has these three definitions with these colors after `VixenSystem.LoadSystemConfig()` runs (`EnsureBuiltInElementTags()`, from VIX-3933, seeds any missing built-in on load without overwriting existing ones).

**Persisting tag changes: when this plan must explicitly save, and when it must not.** `ElementTagCollection`/`ElementTagDefinition` mutations (assigning/removing a tag on a node, or changing a tag definition's `DisplayColor`) only exist in memory until `VixenSystem.SaveSystemConfigAsync()` (`public static async Task<bool> SaveSystemConfigAsync()`, `src\Vixen.Core\Sys\VixenSystem.cs`, line 161) runs and writes `SystemConfig.xml` to disk; conversely, `VixenSystem.ReloadSystemConfig()` (line 334) discards any unsaved in-memory changes by reloading from disk. Two existing UI flows already gate a save behind an explicit user confirmation, and this plan must not duplicate or fight them:

- **Display Setup's OK path**: `VixenApplication.SetupDisplay()` (`src\Vixen.Application\VixenApplication.cs`, lines 990-1008) shows the `DisplaySetup` form modally; on `DialogResult.OK` it calls `await VixenSystem.SaveSystemConfigAsync();` (line 1002); on any other result (Cancel) it calls `VixenSystem.ReloadSystemConfig();` (line 1015) instead, discarding whatever was changed during that Display Setup session.
- **Preview Setup's close path**: the same generic OK-saves/Cancel-reloads shape exists in `VixenApplication.SetupPreviews()` (same file, lines 957-974, wrapping the `ConfigPreviews` list dialog). An individual preview's element/shape setup (`VixenPreviewSetup3`, which hosts the shared `ElementTree` per Milestone 2) is opened from inside that flow via `VixenPreviewModuleInstance.Setup()` (`src\Vixen.Modules\Preview\VixenPreview\VixenPreviewModuleInstance.cs`, line 162), itself invoked from `ConfigPreviews.cs` line 237. Confirm the exact call site that constitutes "the normal close path" during Milestone 2's implementation (this plan's research did not trace a save call inside `VixenPreviewSetup3`/`VixenPreviewModuleInstance` themselves, only the outer `ConfigPreviews` OK/Cancel pair) — but whichever it is, rely on it exactly the way Display Setup's OK/Cancel pair is relied upon below; do not add a second, redundant save.

**Because Milestone 2's tag-assignment context menu lives inside both of those hosts** (it is the one shared `ElementTree` control), a tag assigned or removed there is captured by whichever host's own OK/Cancel-driven save the user performs when they finish — exactly like every other existing `ElementTree` action (rename, delete, group, etc.) already relies on that same mechanism. Milestone 2 must not call `SaveSystemConfigAsync()` itself.

**Milestone 3's Sequencer Tags submenu has no equivalent gated save to rely on** — the Sequencer's own "save" saves the sequence file, a completely different persistence concern from the profile-level `SystemConfig`. Any tag assigned or removed through the Sequencer's Tags submenu must therefore call `await VixenSystem.SaveSystemConfigAsync();` explicitly, immediately after the mutation, so it is not silently lost if the user never happens to open Display Setup or Preview Setup afterward.

**Milestone 8's tag color editor edits `ElementTagDefinition.DisplayColor`, a tag *definition*, not a per-node assignment**, and it is reachable from all three hosts (Display Setup, Preview Setup, and the Sequencer) via each one's Tags submenu. `VixenSystem.SaveSystemConfigAsync()` saves the *entire* `SystemConfig` — every pending change in memory, not just tag data — so it must **not** be called unconditionally from inside the color editor: if the user is mid-edit in Display Setup or Preview Setup (nodes renamed, moved, grouped, etc., not yet confirmed) and opens "Manage Tag Colors..." from there, an unconditional save would silently commit all of that other, still-pending, not-yet-OK'd state too, defeating that host's own Cancel. There is no way to save only the tag color changes and leave the rest of `SystemConfig` untouched — `SaveSystemConfigAsync()` operates on the whole in-memory config, not a subset.

So `ElementTagColorEditorWindow` must be told, at construction, whether *it* is responsible for persisting the change it makes — a generic `bool saveOnClose` constructor parameter (name indicative; finalize during implementation), not a host-specific flag like "invoked from the Sequencer." The window itself only ever needs to know "should I save," not "who opened me" — which caller passes which value is a decision made at each of the three call sites, not something the window needs to branch on by host identity. This keeps the window reusable as-is if a future caller with different save semantics needs it, without having to teach the window about a new host:

- **Callers with no other save path** (the Sequencer's Tags submenu, for now) construct it with `saveOnClose: true`. On Save, the window calls `await VixenSystem.SaveSystemConfigAsync();` explicitly before closing — exactly as before, just driven by the generic flag instead of a Sequencer-specific one.
- **Callers that already have their own OK/Cancel-gated save** (Display Setup's and Preview Setup's Tags submenus, for now) construct it with `saveOnClose: false`. The color change is applied to the in-memory `ElementTagDefinition` (still visible immediately via the same redraw/`Invalidate()` calls as the other case) and is left to be captured — or discarded — by whichever of that host's own OK/Cancel the user performs afterward, exactly like Milestone 2's tag assignments. This means a `Deprecated` color change made from inside Display Setup *is* reverted by a subsequent Display Setup Cancel, along with any other pending changes from that session — this is correct and intended, not a gap, since there is no way to special-case "only the tag color part" out of that Cancel.

This corrects an earlier version of this plan, which had the color editor always call `SaveSystemConfigAsync()` regardless of host; that was wrong because it could silently commit unrelated pending Display Setup/Preview Setup state on an unconditional save, not just the tag color. It also corrects a narrower, immediately-following version that named the flag `invokedFromSequencer` — tying the flag's name to today's one caller that happens to need it would make the window's own contract read as Sequencer-specific, when the actual contract is just "should this window persist its own changes on Save," something any future caller (host or otherwise) can set for itself.
